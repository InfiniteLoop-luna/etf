## 1. 总体结论：建议采用“三表一视图”样本体系

为了兼顾**可追溯、可增量、可回测、可迭代**，我建议不要一开始把所有东西都塞进一张超级宽表，而是采用：

### 建议数据层结构

1. **基础样本表**：`ml_stock_universe_daily`
   - 定义每个交易日、每只股票是否进入训练宇宙
   - 负责样本主键与基础元信息

2. **特征表**：`ml_stock_feature_daily`
   - 存放当日可见特征
   - 每行一个 `trade_date + ts_code`

3. **标签表**：`ml_stock_label_daily`
   - 存放未来收益、方向、回撤、超额收益标签
   - 每行一个 `trade_date + ts_code`

4. **训练视图/导出表**：`ml_stock_sample_daily`
   - 将 universe + features + labels 拼成训练样本
   - 供 Python 直接读取训练

> 这样设计的好处是：
> - 特征可重跑，标签可重跑，互不污染；
> - 新增特征时，不需要重建全部标签；
> - 回测时更容易审计未来函数问题；
> - 后续盘中模型也能复用框架。

## 2. 样本粒度与预测时点

### 2.1 样本粒度
建议粒度：

> **每只股票 × 每个交易日 = 一条样本**

主键：
- `trade_date`
- `ts_code`

### 2.2 预测时点
建议先把首版定义为：

> **T 日收盘后生成特征，预测 T+1 到未来 5/10/20 个交易日的表现**

也就是说：
- 特征使用 `T` 日及以前数据；
- 标签使用 `T+1...T+H` 的未来数据；
- 入场价格默认以 **T 日收盘价 close** 为锚点。

这样和你当前 app 里 `_compute_forward_returns_by_bars(...)` 的思路一致，能减少口径漂移。

### 2.3 为什么先用“收盘后预测”
因为它最稳：
- 数据最全；
- 未来函数最容易控制；
- 与现有 daily / daily_basic / moneyflow / 周月线 / 财务数据天然匹配；
- 后续线上也可以做成“收盘后更新次日观察名单”。

## 3. 训练宇宙（Universe）定义

这一块非常关键。宇宙如果定义不干净，模型会学偏。

### 3.1 建议的基础宇宙
建议 `ml_stock_universe_daily` 纳入：
- 当日存在有效日线价格数据的股票；
- 股票状态为可交易股票（A 股主板 / 创业板 / 科创板 / 北交所可单独标记）；
- 非退市状态；
- 有足够历史长度（如至少 60 个交易日）才进入主样本。

### 3.2 ST 怎么处理
我**不建议在训练阶段直接剔除所有 ST**，更推荐：

#### 方案 A（推荐）
- 保留样本；
- 增加特征：
  - `is_current_st`
  - `has_ever_st`
- 线上展示/推荐时再按策略选择是否过滤。

#### 原因
如果训练时把 ST 全剔掉：
- 会损失真实市场分布；
- 模型学不到“风险票”的边界；
- 后续做风险预测反而不完整。

### 3.3 次新股怎么处理
建议加一层规则：
- `listing_days >= 60` 进入主模型训练；
- `listing_days < 60` 可单独打标，不参与主模型，或者进入低置信度分支。

### 3.4 停牌 / 无流动性处理
建议过滤：
- 当日 `close` 缺失；
- 连续停牌导致未来标签不可计算；
- 流动性极差（如金额极低）可保留但打 `low_liquidity_flag`。

### 4.4 `ml_stock_sample_daily`

作用：
- 最终训练样本视图或物化表；
- 联表后给 Python 模型直接读取。

#### 逻辑
```sql
SELECT
  u.*,
  f.*,
  l.*
FROM ml_stock_universe_daily u
JOIN ml_stock_feature_daily f
  ON u.trade_date = f.trade_date AND u.ts_code = f.ts_code
JOIN ml_stock_label_daily l
  ON u.trade_date = l.trade_date AND u.ts_code = l.ts_code
WHERE u.sample_eligible = TRUE;
```

## 5. 标签口径定义（必须统一）

这一节是最关键的，后面代码必须按这里实现。

---

### 5.1 入场价格定义

首版统一使用：

> `entry_price = T 日收盘价 close`

原因：
- 与当前 app 一致；
- 与收盘后跑模型一致；
- 数据最稳定。

后续如果做盘中模型，再增加：
- `entry_basis = same_day_14_30_price`
- 或 `next_open`

但首版先不要混口径。

---

### 5.2 未来收益标签

定义：

> `ret_fwd_H = close(T+H) / close(T) - 1`

其中 `H ∈ {1, 3, 5, 10, 20}`。

#### SQL/Python 逻辑建议
- 按 `ts_code` 分组；
- 按 `trade_date` 升序；
- 使用 `shift(-H)` 取得未来第 H 个交易日收盘价；
- 计算收益率。

#### 注意
这里用的是“**交易日 bar 偏移**”，不是自然日。

---

### 5.3 方向标签

定义：

> `y_up_H = 1[ret_fwd_H > 0]`

可再扩展：
- `y_up_strong_5d = 1[ret_fwd_5d > 0.03]`
- `y_breakout_5d = 1[max_upside_fwd_5d > 0.05]`

但 V1 先保持简单。

---

### 5.4 最大回撤标签

定义：

> `max_dd_fwd_H = min(low(T+1...T+H)) / close(T) - 1`

如果 `H=5`：
- 从未来 5 个交易日的最低价中取最小；
- 与 `T` 日收盘价比较。

这比只看终点收益更扎实，因为它能反映：
- 胜率之外的持仓痛苦程度；
- 趋势虽对但中间是否大回撤。

### 5.5 最大上行空间标签

定义：

> `max_upside_fwd_H = max(high(T+1...T+H)) / close(T) - 1`

用途：
- 评估“有没有冲高机会”；
- 适合短线机会筛选。

---

### 5.6 相对收益标签（强烈推荐）

因为大盘/行业环境对绝对收益影响很大，我建议同步生成超额收益标签。

#### 方案 A：相对行业等权收益

> `alpha_fwd_H_vs_industry = ret_fwd_H - industry_ret_fwd_H`

其中：
- `industry_ret_fwd_H` = 同行业样本在当日的等权平均未来收益

#### 方案 B：相对市场等权收益

> `alpha_fwd_H_vs_market = ret_fwd_H - market_ret_fwd_H`

其中：
- `market_ret_fwd_H` = 全市场等权平均未来收益

#### 为什么推荐这个标签
模型学绝对收益时，经常只是学到“大盘牛市”或“热点普涨”；
学超额收益时，更容易学到：
- 个股是否更强；
- 相对同行是否更突出；
- 真正的选股能力。

## 6. 行业基准与市场基准的推荐实现

### 6.1 行业基准
建议用：
- 同日、同行业、`sample_eligible = true` 股票的等权未来收益均值

优点：
- 不依赖外部行业指数；
- 完全可以用现有数据内部构造；
- 与训练宇宙口径一致。

### 6.2 市场基准
建议 V1 用：
- 全市场等权未来收益均值

如果后续指数数据更完整，可加：
- 沪深300 / 中证1000 / 创业板指 等市场风格基准。

## 7. 特征生成的时间对齐规则（必须严格）

---

### 7.1 日线价格 / 成交 / 估值
这些字段天然是 `trade_date` 对齐。
规则：
- 使用 `T` 日收盘后已知值；
- 滚动特征只用 `<= T` 历史。

### 7.2 财务数据
必须使用：

> `ann_date <= trade_date` 的最近一期财务记录

实现方式：
- 对每个 `trade_date, ts_code` 做 as-of join；
- 可在 SQL 里用 `DISTINCT ON` / `ROW_NUMBER` 按 `ann_date DESC` 取最近一期。

### 7.3 周/月线数据
必须确认：
- `vw_ts_stk_week_month_adj` 在日级对齐时是否有前向填充；
- 若有，确保填充的是“当前日已知的最近周/月状态”，不是未来周末/月末值提前泄漏。

### 7.4 资金流与事件数据
建议规则：
- 收盘后模型可以使用当日全量资金流与事件结果；
- 如果某些数据源当天可能延迟，则加：
  - `has_moneyflow_feature`
  - `has_event_feature`
  - `feature_complete_ratio`

## 9. 回测与训练切分方案

这部分也要写死，不然后面容易“看起来很准”。

### 9.1 禁止随机切分
不能用随机 train/test split。

必须使用：
> **时间序列滚动切分（walk-forward）**

### 9.2 推荐切分方式

#### 方案 A：年度滚动
- Train: 2022-2024
- Valid: 2025H1
- Test: 2025H2
- Forward: 2026

#### 方案 B：滑窗滚动（更推荐）
- 每次训练用过去 24 个月
- 验证用接下来 3 个月
- 测试用再接下来 3 个月
- 向前滚动 1 个月重复一次

### 9.3 为什么用滑窗
因为 A 股风格切换快：
- 牛市、熊市、震荡市特征分布不同；
- 固定训练集容易过时；
- 滑窗更接近真实线上使用场景。

## 11. 数据质量校验（上线前必须有）

我建议每次样本生成都输出 QC（质量检查）结果。

### 11.1 Universe 层 QC
- 每日样本数
- 各行业样本数
- `sample_eligible` 比例
- 次新/ST/低流动性占比

### 11.2 Feature 层 QC
- 每列缺失率
- 极值统计（P1/P99）
- 数值溢出/异常值占比
- 横截面排名字段是否完整

### 11.3 Label 层 QC
- 各 horizon 标签覆盖率
- `ret_fwd_H` 均值/方差
- `max_dd_fwd_H` 分布
- 行业/市场 alpha 分布

### 11.4 泄漏审计
- 财务特征：随机抽样核对 `ann_date <= trade_date`
- 周/月特征：抽样检查周中是否拿到未来周末值
- 资金流数据：检查是否存在未来回填

这个 QC 体系一旦没有，后面模型结果很难信，我不想让我们踩这个坑。
