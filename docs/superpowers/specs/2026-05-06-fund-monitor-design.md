# 基金监测设计文档

日期：2026-05-06  
状态：已在对话中完成方向确认  
范围：基于 `新增证券分析表单-20260505.xlsx` 的 `公募&私募基金` sheet，在 Web 页面 `宏观` 分类下新增 `📈 基金监测` 页面，并支持长期导入与手工维护。

## 1. 背景

当前应用在 `宏观` 分类下已经逐步扩展出可维护的数据型页面，例如：

- `🌏 宏观经济`
- `🏦 本外币存款`
- `📊 指数监测`

参考工作簿 `新增证券分析表单-20260505.xlsx` 中还有一个 `公募&私募基金` sheet，包含公募基金、私募证券投资基金、私募资管权益基金以及权益类理财产品等多组月度数据。用户希望继续把这一组数据纳入 Web 页面，并放在 `宏观` 下面统一管理和展示。

这张 sheet 的结构不是单月单表，而是：

1. 顶部存在一组最新月摘要
2. 下方按月份纵向重复数据区块
3. 每个区块内部又按：
   - `当月情况`
   - `环比变动情况`
   - `同比变动情况`
   组织字段

因此，该数据源更适合设计为一个“基金监测分析页”，而不是把 Excel 原样平铺到网页上。

## 2. 用户已确认的设计方向

对话中已确认选择：

- 页面位置放在 `宏观` 下面
- 页面形态采用一个总览型页面，而不是拆成多个独立页面
- 推荐页面名称使用 `📈 基金监测`
- 页面不仅展示公募基金，还要同时利用私募与理财相关数据

本次设计默认采用这一方向继续。

## 3. 目标

- 在 `宏观` 分类下新增 `📈 基金监测` 页面
- 将 `公募&私募基金` sheet 转为适合数据库存储和 Web 分析的月度结构化数据
- 提供一个可读性强的总览型仪表盘，用于观察基金资金结构和风险偏好变化
- 支持后续新月份数据继续导入
- 支持必要的手工补录和修订

## 4. 非目标

首版不做以下内容：

- 对基金产品逐只明细穿透
- 与基金持仓热股、ETF、个股行情做交叉归因
- 自动生成研判结论或自然语言摘要
- 接入外部 API 自动更新该数据

## 5. 数据特征与业务理解

### 5.1 核心分组

从工作簿可以识别出以下主要分类：

- `封闭式基金`
- `开放式基金`
- `其中：股票基金`
- `其中：混合基金`
- `其中：债券基金`
- `其中：货币基金`
- `其中：QDII基金`
- `合计`
- `私募证券投资基金`
- `私募资管权益基金`
- `权益类理财产品`

### 5.2 核心指标

每个分类下主要字段为：

- `基金数量（只）`
- `份额（亿份）`
- `净值（亿元）`
- `单位净值（元）`

并同时给出：

- 当月值
- 环比变动值
- 同比变动值

### 5.3 使用上的现实约束

这张表中并不是每个分类都完整覆盖四个指标：

- 公募基金分类通常字段较完整
- 私募和理财类字段更偏向 `净值（亿元）`
- 部分 `单位净值` 单元格存在 `#DIV/0!`
- 部分字段天然为空值

因此系统设计必须允许：

- 空值
- Excel 错误值转空
- 不同分类字段覆盖不完全

## 6. 页面挂载与信息架构

### 6.1 导航位置

在 `宏观` 二级页面中新增：

- `📈 基金监测`

并保证桌面端与 iPhone 模式都能访问到该页面。

### 6.2 页面整体结构

页面采用“总览优先、维护其次”的布局，建议分为四个主要区域：

1. 顶部筛选与操作区
2. KPI 总览区
3. 主图分析区
4. 快照表与维护区

该页面重点是帮助用户快速判断：

- 公募整体规模是否扩张
- 股票/混合/债券/货币/QDII 的结构如何变化
- 私募与权益理财是否同步增强或减弱
- 当前是否体现出更强的风险偏好

## 7. 首页布局设计

### 7.1 顶部筛选与操作区

建议提供以下控件：

- 月份选择
- 分类多选
- 维度切换：
  - `净值`
  - `份额`
  - `基金数量`
  - `单位净值`
- 时间范围：
  - `最近12个月`
  - `最近24个月`
  - `全部`
- `新增月份`
- `单分类补录/修改`
- `批量导入 Excel`

设计原则：

- 默认先服务“读数据”
- 维护入口可见，但不压住首屏分析内容

### 7.2 KPI 总览区

顶部 KPI 推荐展示：

- 最新月份
- 公募合计净值
- 股票基金净值
- 混合基金净值
- 私募证券投资基金净值
- 权益类理财产品净值

补充规则：

- KPI 默认使用最新月份
- 若某项当月为空，则显示 `-`
- 私募/理财类 KPI 以 `净值（亿元）` 为主，不强依赖份额或单位净值

### 7.3 主图分析区

首页中部推荐做三张主图：

#### 图 1：基金净值总趋势

用于对比以下序列的月度变化：

- 公募证券投资基金合计
- 私募证券投资基金
- 私募资管权益基金
- 权益类理财产品

用途：

- 快速观察场内外权益资金整体扩张或收缩

#### 图 2：公募结构趋势

用于展示以下公募结构序列：

- 股票基金
- 混合基金
- 债券基金
- 货币基金
- QDII基金

建议默认展示 `净值（亿元）`，并允许切换到：

- 份额
- 基金数量

用途：

- 观察公募内部风格切换和结构变化

#### 图 3：同比 / 环比变化曲线

支持用户选择一个指标维度，例如：

- 净值
- 份额
- 基金数量
- 单位净值

再对所选分类绘制：

- 环比变化曲线
- 同比变化曲线

用途：

- 回答“哪些类别正在改善，哪些类别在走弱”

### 7.4 快照表区

页面下方提供最新月份快照表，一行一个分类。

建议默认列为：

- 分类名称
- 基金数量
- 份额（亿份）
- 净值（亿元）
- 单位净值（元）
- 环比基金数量变动
- 环比份额变动
- 环比净值变动
- 同比基金数量变动
- 同比份额变动
- 同比净值变动

表头全部使用中文。

### 7.5 维护区

维护区建议与 `指数监测` 保持一致，支持：

- 按月份批量录入
- 单分类补录 / 修改
- Excel 导入预览

这样未来新增月份时，维护体验一致，不会每个页面都长得不一样。

## 8. 数据模型设计

### 8.1 建模原则

不要保留 Excel 的横向分组排版作为数据库主结构。  
应标准化为“一条记录 = 一个月份 + 一个分类”。

### 8.2 推荐逻辑字段

#### 主键与标识字段

- `month`
- `category_name`

#### 分类辅助字段

- `category_group`
  - 可取：`public_fund` / `private_fund` / `wealth_mgmt`
- `category_level`
  - 可取：`total` / `primary` / `subtype`
- `sort_order`

#### 当月情况

- `fund_count`
- `share_amount`
- `nav_amount`
- `unit_nav`

#### 环比变动情况

- `mom_fund_count`
- `mom_share_amount`
- `mom_nav_amount`
- `mom_unit_nav`

#### 同比变动情况

- `yoy_fund_count`
- `yoy_share_amount`
- `yoy_nav_amount`
- `yoy_unit_nav`

#### 审计字段

- `source_type`
- `source_file`
- `created_at`
- `updated_at`

### 8.3 分类映射建议

建议预置一套分类映射规则，例如：

- `其中：股票基金` -> `public_fund` / `subtype`
- `其中：混合基金` -> `public_fund` / `subtype`
- `其中：债券基金` -> `public_fund` / `subtype`
- `其中：货币基金` -> `public_fund` / `subtype`
- `其中：QDII基金` -> `public_fund` / `subtype`
- `合计` -> `public_fund` / `total`
- `私募证券投资基金` -> `private_fund` / `primary`
- `私募资管权益基金` -> `private_fund` / `primary`
- `权益类理财产品` -> `wealth_mgmt` / `primary`

该映射主要服务于前端默认分组展示和排序。

## 9. 存储方案

首版应直接使用 PostgreSQL 存储，不把 Excel 作为系统主存储。

理由：

- 当前项目已经大量使用 PostgreSQL
- 月度数据更适合做结构化查询、覆盖写入和趋势展示
- 导入、更新、纠错都更稳定

数据库连接方式应继续复用当前项目已有的 `ETF_PG_*` 约定，不新增一套独立配置方式。

建议表名示例：

- `macro_fund_monitor_monthly`

主键建议：

- 唯一键：`(month, category_name)`

## 10. 导入解析设计

### 10.1 首版导入来源

首版仅要求兼容 `新增证券分析表单-20260505.xlsx` 中的 `公募&私募基金` sheet。

### 10.2 解析原则

解析器应按“纵向月份区块”识别数据，而不是只读顶部摘要。

每个区块的特征包括：

- 区块起始附近出现月份
- 同一组分类按固定顺序重复出现
- 指标列结构固定

解析后应输出：

- 一个月份
- 一个分类
- 一行结构化记录

### 10.3 错误值处理

解析器需特别处理：

- `#DIV/0!` -> `None`
- 空字符串 -> `None`
- 不可转数字 -> `None`

### 10.4 导入预览

写入前必须展示预览，并区分：

- 新增记录
- 覆盖记录

首版不强求做失败记录专区，只要解析异常能明确报错即可。

## 11. 手工维护设计

### 11.1 新增月份

允许用户新增一个月份，并按多个分类批量填写。

交互形式建议：

- `st.data_editor`

优点：

- 更接近 Excel 录入习惯
- 适合月度批量补录

### 11.2 单分类补录 / 修改

允许用户：

- 先选择月份
- 再选择分类
- 打开该分类的详细字段表单进行修正

这样适合修复单个分类的异常值。

## 12. 图表与展示规则

### 12.1 默认展示重点

首版页面默认应该强调：

- `净值（亿元）`

因为在这张表里，净值对公募、私募和理财几类数据的可用性最好，也最适合横向比较。

### 12.2 分类默认勾选建议

进入页面时默认勾选：

- `合计`
- `其中：股票基金`
- `其中：混合基金`
- `私募证券投资基金`
- `私募资管权益基金`

这样既能看公募主线，也能看私募补充信息。

### 12.3 缺失值展示

规则如下：

- 图表中自动跳过空值
- 表格中显示为空或 `-`
- KPI 缺失时显示 `-`

## 13. 测试与验证要求

首版至少覆盖：

- 解析器能正确读取多个月份区块
- 能处理 Excel 错误值和空值
- 导入预览能正确区分新增与覆盖
- 页面导航能正确挂到 `宏观`
- 核心趋势构造函数输出形状正确

## 14. 实施建议

建议沿用 `指数监测` 的实现模式，拆成三部分：

1. `fund_monitor_importer`
   - 负责解析工作簿
2. `fund_monitor_store`
   - 负责建表、读写、趋势数据拼装
3. `app.py` 页面集成
   - 负责导航、仪表盘与维护交互

这样可以最大限度复用已经验证过的页面和存储模式。

## 15. 总结

`公募&私募基金` sheet 最适合被转化为一个“基金监测”总览页，而不是照搬 Excel 格式。  
页面放在 `宏观 / 📈 基金监测` 下，首版以净值趋势、结构趋势和同比环比变化为核心，同时保留导入与手工维护能力。  
数据层统一落 PostgreSQL，继续沿用当前项目的环境变量配置与月度长表模式。
